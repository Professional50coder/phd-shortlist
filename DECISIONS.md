# DECISIONS

## 1. Architecture overview

This pipeline is built as a six-stage shortlist generator:

1. **Profile Parse** (`profile_parser.py`)
   - Extracts hard constraints from the student profile.
   - Builds a conservative set of search keywords from research interests.
   - Enforces `target_countries` and intake data before any API work.

2. **Source Fetch** (`source_fetcher.py`)
   - Uses Semantic Scholar, OpenAlex, NIH Reporter, and UKRI.
   - Fetches broadly and caches all API responses to disk.
   - Returns raw candidate records with paper/grant evidence.

3. **PI Filter** (`pi_filter.py`)
   - Applies the contamination firewall.
   - Rejects wrong country, trainees from personal fellowships, early-career false positives, and wrong-domain candidates.
   - Flags high-collision names without hard-rejecting them.

4. **Rank & Tier** (`ranker.py`)
   - Scores candidates by domain match, career-stage confidence, recency, and publication signal.
   - Assigns `reach`, `target`, or `safety` tiers.
   - Enforces area coverage by seeding top candidates across the student's stated interests.

5. **Personalise** (`personaliser.py`)
   - Generates `why_match` text for each shortlist entry.
   - Uses a local template fallback if an LLM is unavailable.

6. **Output JSON** (`output_schema.py`)
   - Validates final output against a typed schema.
   - Ensures all required fields are present and country codes are ISO-3166 alpha-2.


## 2. Key grading priorities and design decisions

### 2.1 Contamination first

The pipeline intentionally prefers precision over raw quantity. The filter stage rejects any candidate with weak or ambiguous evidence rather than passing them through on keyword overlap alone.

- **Country adherence** is a hard constraint. If the candidate's inferred country is outside `target_countries`, they are rejected before ranking.
- **Top 30 contamination** is treated as more important than having a long list. The ranking stage is conservative about tiering and penalises high-collision names.
- **Coverage is second**: the ranker seeds one candidate per research interest to avoid narrow lists.

### 2.2 Career-stage detection

This is the hardest failure mode, so we use multiple independent signals:

- explicit titles (`Professor`, `Lecturer`, `PhD Student`, `Postdoc`)
- `h-index`
- publication span
- paper count

Thresholds:
- `MIN_H_INDEX = 5`
- `MIN_PUB_SPAN_YEARS = 4`
- `MIN_PAPER_COUNT = 8`

Trade-off: a hard threshold would exclude rising-star assistant professors in low-citation fields. So we use score aggregation and only reject when most signals fail.

### 2.3 Wrong-domain leakage

A two-stage check is used:

1. **Embedding cosine similarity** between candidate evidence and student research text.
   - `hard_pass >= 0.52`
   - `hard_fail < 0.32`
2. **LLM binary classifier** for borderline cases.

Trade-off: embeddings handle broad false positives cheaply, and the LLM is reserved for ambiguous candidates only.

### 2.4 Name collisions

High-collision surnames are flagged and deprioritised rather than rejected.

Identity fingerprint uses:
- normalized name
- normalized institution
- area-relevant evidence
- country hint

For surnames like `Wang`, `Singh`, `Smith`, the pipeline requires at least two corroborating signals or it marks the candidate as `high_collision_name`.

### 2.5 Fellowship eligibility

NIH and UKRI grants are treated carefully:
- F31/F32/K99/MSCA-PD are personal fellowships and often list trainees.
- If a candidate record consists only of these fellowship grants and no paper evidence, they are rejected.

This avoids the hard fail of surfacing a trainee instead of a supervisor.


## 3. Module-specific decisions

### `profile_parser.py`
- Enforces hard constraints early.
- Expands research interests using a controlled synonym dictionary.
- Infers `fully_funded_only` conservatively.

### `source_fetcher.py`
- Caches all API responses in `.cache/fetch` to make runs reproducible and debuggable.
- Uses `aiohttp` for parallel source fetching.
- Over-fetches intentionally; precision is enforced downstream.

### `pi_filter.py`
- The filter order is chosen for efficiency:
  1. country + fellowship rules
  2. career-stage
  3. domain
  4. identity
- Rejections are logged in `rejection_log.json` for DECISIONS examples.

### `ranker.py`
- Domain similarity and career-stage confidence are the primary score drivers.
- Recency, h-index, and evidence count provide secondary differentiation.
- High-collision names are penalised so they are less likely to appear in top tiers.

### `personaliser.py`
- The pipeline supports LLM generation when `OPENAI_API_KEY` is set.
- If no key is available, text is generated by a deterministic template.
- This ensures the pipeline remains functional without external calls.

### `output_schema.py`
- Uses Pydantic validation so invalid output fails loudly.
- Ensures every shortlist entry has a valid ISO country code and tier.


## 4. Trade-offs documented here

- **H-index threshold** is not a hard reject. It is one signal among several because early-career faculty may have lower h-indices.
- **Domain similarity thresholds** are intentionally conservative to reduce false positives. Some true matches may be delayed into the review set rather than incorrectly included.
- **High-collision names** are not hard rejected because the worst mistake is excluding a real PI. Instead, they are downgraded and surfaced in the safety tier.
- **Email extraction** is not implemented by guessing; the pipeline preserves `null` rather than inventing unreliable contact details.
